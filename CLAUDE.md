# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Contexto y seguimiento de tareas

- **Siempre leer `PLANNING.md`** al inicio de una nueva conversaciГіn para entender la arquitectura, el stack y el estado actual del proyecto.
- **Consultar `TASK.md`** antes de empezar una tarea. Si la tarea no estГЎ listada, agregarla con fecha. Marcarla completada en cuanto termine.
- **Workflow PRP**: para features no triviales, usar `/generate-prp INITIAL.md` (investiga + genera el PRP) y luego `/execute-prp PRPs/<feature>.md` (implementa con puertas de validaciГіn).

## Proyecto

App Mobile de Asistencia IA que integra agentes de IA gratuitos. TypeScript + pnpm, diseГұada con Clean Architecture y desarrollada con TDD (Spec-First). El nГәcleo (`domain` + `application` + adapters) se construyГі y probГі **sin framework** hasta tener cobertura verde; reciГ©n entonces se instalГі **React Native / Expo** (Expo SDK 54, RN 0.81, React 19.1) como capa de UI. El nГәcleo sigue siendo independiente del framework: la lГіgica vive en view-models de TS puro testeables con Vitest, y RN solo renderiza.

## Comandos

Gestor de paquetes: **pnpm** (vГӯa corepack, versiГіn fijada en `packageManager`). No usar npm/yarn вҖ” `.npmrc` impone `node-linker=isolated` para evitar dependencias fantasma.

```bash

pnpm web                  # expo start --web (bundle web, sin emulador)
pnpm android / pnpm ios   # abrir en emulador/simulador
pnpm test                 # vitest run (toda la suite, una vez)
pnpm test:watch           # modo watch para el ciclo REDвҶ’GREENвҶ’REFACTOR
pnpm test:domain          # solo tests/domain
pnpm test:application     # solo tests/application
pnpm test:contracts       # solo tests/contracts
pnpm test:infrastructure  # solo tests/infrastructure
pnpm test:coverage        # con cobertura v8 (umbral 80% en domain/application)
pnpm typecheck            # tsc --noEmit (forma parte del "verde": debe pasar siempre)
pnpm lint                 # eslint, --max-warnings=0 (hace cumplir la frontera de capas)
pnpm build                # tsc -p tsconfig.build.json вҶ’ dist/ (excluye tests y dominio puro de tipos)
pnpm format               # prettier --write
```

Ejecutar **un solo archivo o un test concreto**:

```bash

pnpm exec vitest run -t "no permite agregar mensajes a una conversacion cerrada"
```

## Arquitectura

Capas con la **regla de dependencia hacia adentro** (`infrastructure вҶ’ adapters вҶ’ application вҶ’ domain`). El sentido de los imports lo hace cumplir ESLint (`no-restricted-imports` en `eslint.config.js`): `src/domain/**` no puede importar de ninguna otra capa, y `src/application/**` no puede importar de `adapters` ni `infrastructure`. Si un import cruza una frontera prohibida, `pnpm lint` falla вҖ” no es solo convenciГіn.

- **`src/domain/`** вҖ” Entities y Value Objects. LГіgica pura, sin I/O ni dependencias externas. Las entidades usan **constructor privado + factory estГЎtica** (`Message.create()`, `Conversation.start()`), son inmutables hacia afuera, se autovalidan en construcciГіn y devuelven copias defensivas (ej. `Conversation.history`). Las invariantes de negocio se rompen lanzando un error que extiende `DomainError` (ej. `ConversationClosedError`).
- **`src/application/`** вҖ” Use Cases y **Ports** (interfaces). El use-case recibe los Ports por constructor (DIP) y orquesta el flujo. Los Ports expresan el *quГ©* necesita la app (ej. `AssistantAgentPort`), nunca el *cГіmo*.
- **`src/adapters/`** вҖ” Implementaciones concretas: `ai-agents/` (`OllamaLocalAgentAdapter` local sin auth y `HuggingFaceAgentAdapter` con Bearer token; comparten el `FetchLike` inyectable de `ai-agents/http.ts`; cada uno mapea su DTO crudo вҖ”forma distintaвҖ” al dominio con Zod y traduce fallos a `AgentUnavailableError`), `persistence/` (`InMemoryConversationRepo` para tests + `AsyncStorageConversationRepo` real, que serializa a un DTO Zod y depende de una interfaz `KeyValueStorage` inyectable, no de AsyncStorage directo), `ui/`. `di/createAssistantAgents.ts` construye un agente por proveedor (record exhaustivo sobre `AiAgentProvider`) y `RoutingAssistantAgent` (adapter genГ©rico) enruta a uno segГәn el proveedor seleccionado, cambiable **en runtime** vГӯa `select()`. El proveedor inicial es `AI_AGENT_PROVIDER`. El container expone un `AgentSelector` a la UI; el contrato garantiza que todos los agentes se comportan igual. La UI usa **view-models framework-agnostic** (`ChatViewModel`): un presenter en TS puro con patrГіn store (`getState()` + `subscribe()`) que mapea las entidades a DTOs de presentaciГіn. Un hook de React/RN lo envolverГЎ con `useSyncExternalStore` cuando se instale Expo вҖ” la lГіgica de UI es testeable con Vitest sin renderizar.
- **`src/infrastructure/`** вҖ” Composition Root: Гәnico lugar donde se instancian clases concretas y se inyectan (`di/container.ts`). `config/env.ts` es el **Гәnico** sitio que lee `process.env` y valida con Zod.
- **`src/shared/`** вҖ” Transversal. `Result<T,E>` (`result/Result.ts`) para errores explГӯcitos sin `throw` en application/adapters, y la jerarquГӯa de errores tipados (`errors/`).

### ConvenciГіn de errores

Dos mecanismos coexisten a propГіsito: el **dominio lanza** (`throw`) subclases de `DomainError` para violar invariantes; **application y adapters devuelven `Result<T, E>`** en lugar de lanzar, de modo que el camino de error queda en la firma del tipo.

## Tests (TDD)

`tests/` estГЎ separado por nivel, espejando las capas:
- `tests/domain/` вҖ” unitarios puros.
- `tests/application/` вҖ” aceptaciГіn de use-cases con **Fakes/Stubs** de los Ports (sin I/O real).
- `tests/contracts/` вҖ” un mismo contrato (`*.contract.ts`) corre contra cada adaptador de un Port para garantizar que todos lo cumplen igual (IA y persistencia). El archivo `*.contract.ts` es autocontenido: define la funciГіn de contrato y la invoca con la factory de cada adaptador al final del archivo (debe contener `describe`, no solo exportar).
- `tests/builders/` вҖ” **Test Data Builders** (`MessageBuilder.aMessage().withText(...).build()`). Crear datos de prueba siempre vГӯa Builders, no construyendo entidades a mano.
- `tests/fakes/` вҖ” **Fakes reutilizables** de los Ports (ej. `FakeAssistantAgent`), con factories por escenario (`thatReplies(...)`, `thatIsUnavailable()`).
- `tests/infrastructure/` вҖ” Composition Root. `loadEnv()` se prueba con una fuente de env inyectada (no `process.env` real); `createContainer()` verifica que el grafo queda cableado.
- `tests/adapters/` вҖ” adaptadores sin contrato de Port, ej. los view-models de UI (`ChatViewModel`) probados como lГіgica pura con Vitest.

Glob de Vitest: `tests/**/*.{spec,test,contract}.ts`. Los alias de import (`@domain/*`, `@application/*`, `@adapters/*`, `@infrastructure/*`, `@shared/*`) se definen una sola vez en `tsconfig.json` y Vitest los lee vГӯa `vite-tsconfig-paths` вҖ” no duplicar alias en `vitest.config.ts`.

Flujo: escribir el test que falla (RED) вҶ’ mГӯnimo cГіdigo para pasar (GREEN) вҶ’ refactor, subiendo de capa hacia afuera. `pnpm typecheck` cuenta como parte del verde.

## Expo / React Native

El toolchain de Expo (Metro/Babel) impuso decisiones que conviene conocer:

- **SDK fijado en 54 por Expo Go (iOS)**: en iPhone **fГӯsico**, el Expo Go del App Store soporta como mГЎximo **SDK 54** (SDK 55/56 requieren `eas go`/TestFlight con cuenta Apple paga). Por eso el proyecto estГЎ en SDK 54 aunque exista 56. Si se sube de SDK, deja de abrir en Expo Go estГЎndar.
- **`expo-status-bar` NO es config plugin**: es un componente que se importa, no va en `app.json > plugins`. Ponerlo ahГӯ crashea el arranque en SDK 54 (`Unable to resolve a valid config plugin for expo-status-bar`). En `plugins` solo van plugins reales (ej. `expo-font`).
- **`package.json` NO tiene `"type": "module"`** вҖ” los configs de Expo (`babel.config.js`, `metro.config.js`) son CommonJS. Por eso el flat config de ESLint es `eslint.config.mjs` y el de Vitest es `vitest.config.mts` (necesitan cargarse como ESM).
- **pnpm `isolated` + build tooling**: el ecosistema RN asume `node_modules` aplanado. Mantenemos `node-linker=isolated` a propГіsito; cuando Metro/Babel falle con `MODULE_NOT_FOUND` por un preset/plugin (ej. `babel-preset-expo`), la soluciГіn es **declararlo como devDependency explГӯcita** (es justo lo que pide el principio anti-phantom), NO cambiar a `hoisted`.
- **Alias en Metro**: Expo lee los `paths` de `tsconfig.json` automГЎticamente вҖ” no duplicar alias en `metro.config.js`.
- **Entry point**: `index.js` вҶ’ `registerRootComponent(App)`. `src/infrastructure/app/App.tsx` es el Composition Root del runtime: `createContainer(loadEnv(expoEnvSource()), AsyncStorage)` y arma el `ChatViewModel`. `createContainer` recibe el `KeyValueStorage` por parГЎmetro: el runtime pasa AsyncStorage, los tests un fake вҖ” asГӯ el container es testeable en Node sin el mГіdulo nativo.
- **CachГ© de Metro en Windows/OneDrive**: el crawler incremental puede fallar con `EINVAL: readlink` sobre symlinks de pnpm al aplicar el delta de cachГ©. SoluciГіn: `expo start --clear` / `expo export --clear` (fuerza crawl limpio; el bundle funciona bien).
- **Env en RN**: no hay `.env` en runtime; `expoEnvSource()` (en `config/env.ts`) lee las vars `EXPO_PUBLIC_*` que Babel inyecta en build, con defaults locales. `process.env` sigue confinado a `config/env.ts`. `loadEnv` valida con Zod, incluido un `superRefine` que exige `AI_AGENT_API_KEY` cuando `AI_AGENT_PROVIDER=huggingface`. TambiГ©n expone la config de transcripciГіn (`AI_STT_BASE_URL` / `AI_STT_MODEL` y `AI_STT_API_KEY` opcional que en `expoEnvSource()` cae al token del agente). **El default de `AI_STT_BASE_URL` es `https://router.huggingface.co/hf-inference/models`** (NO el viejo `api-inference.huggingface.co`, que HuggingFace dio de baja: tiraba `Network request failed` aunque el chat вҖ”que ya usa `router.huggingface.co`вҖ” funcionara). El adapter arma `${baseUrl}/${model}` вҶ’ `вҖҰ/hf-inference/models/openai/whisper-large-v3`. `App.tsx` envuelve la creaciГіn del container en un `try/catch` y muestra una pantalla "Error de configuraciГіn" en vez de crashear si la env es invГЎlida. Esa pantalla incluye un botГіn **"Borrar ajustes guardados y reintentar"** (red de seguridad anti-brick): borra `settings:apiKey` + `settings:ollamaBaseUrl` de AsyncStorage y reinicia (`bootKey++`) con los defaults del build, sin reinstalar. Es la **Гәnica vГӯa de vuelta** si un override guardado hace fallar `loadEnv`. Para que ese caso casi no ocurra, `SettingsViewModel.save()` **valida la URL de Ollama antes de persistir** con una regla mГЎs estricta que env (`z.string().url().refine(http(s)://)`): Zod `.url()` por sГӯ solo acepta `localhost:11434` (esquema `localhost:`), que pasarГӯa env pero el adapter no puede usar; URL invГЎlida вҶ’ `phase:'error'` con mensaje inline y **no toca persistencia** (cubierto por tests).
- **`.env` vs `.env.local` (Expo SDK 56)**: `@expo/env` 2.x **rechaza** variables que no empiezan con `EXPO_PUBLIC_` dentro de `.env` (`Refused to load personal environment variables from a non-.local env file`). Las "personales" (ej. `REACT_NATIVE_PACKAGER_HOSTNAME` para forzar la IP del QR en LAN, secretos) van en **`.env.local`**; en `.env` solo `EXPO_PUBLIC_*`. Ambos estГЎn gitignoreados.
- **Borrar `dist-web` (export web) en Windows**: las rutas de assets superan MAX_PATH y `Remove-Item` falla. Usar `robocopy <dir-vacГӯo> dist-web /MIR` y luego borrar. `dist-web/**` estГЎ en `.gitignore` y en los `ignores` de ESLint.
- **Teclado / detalles de chat**: el chat usa `KeyboardAvoidingView` con `useHeaderHeight()` (de `@react-navigation/elements`) para que el input suba sobre el teclado en iOS. El **safe-area inferior** lo aplica la fila de input vГӯa `useSafeAreaInsets()` вҶ’ `marginBottom: Math.max(insets.bottom, spacing.lg)` (el `KeyboardAvoidingView` no reserva el home indicator y los botones quedaban cortados con el teclado cerrado); el `content` ya no usa `paddingBottom` (lo maneja la fila) y el FAB вҶ“ calcula su `bottom` con el mismo inset. Las burbujas muestran hora (`Message.createdAt` вҶ’ `ChatMessageVM.time`) y botГіn "Copiar" (`expo-clipboard`); auto-scroll al Гәltimo mensaje. **EnvГӯo con UI optimista**: `ChatViewModel.send()` arma una burbuja provisoria del usuario y la muestra **al instante** (status `sending` + "La IA estГЎ pensandoвҖҰ") en vez de esperar el round-trip; al terminar, el snapshot real del `conversation.history` (user + assistant) la reemplaza. En error, el snapshot conserva la pregunta. EstГЎ cubierto por test (el primer cambio a `sending` ya trae el mensaje del usuario). Cada burbuja entra con una animaciГіn **one-shot** (`AnimatedBubble`): fade + slide-up + **spring con leve overshoot de scale** (0.94вҶ’1.0 con un pop hasta ~1.03 en el overshoot, asienta en 1.0 vГӯa `interpolate` con `inputRange:[0,1,1.15]`) вҖ” un "pop" sutil, mГЎs vivo que un timing lineal. Corre una vez al montar (keyExtractor estable вҶ’ no re-anima en cada render), `useNativeDriver:false` para compat web como el drawer. La burbuja "pensando" tambiГ©n entra animada y su indicador son **`TypingDots`** (tres puntos que rebotan en secuencia, el clГЎsico "escribiendoвҖҰ"): es un `Animated.loop` **acotado** que solo vive mientras la burbuja estГЎ montada (durante el request) y se **detiene en el cleanup** al desmontar вҖ” no es un loop siempre-activo (esos saturan el render en web). ReemplazГі al `ActivityIndicator` estГЎtico (que sigue usГЎndose en el estado "TranscribiendoвҖҰ"). Los botones del `inputRow` (рҹ“Һ рҹ“· рҹҺӨ Enviar) usan **`PressableScale`** (componente reutilizable): hace un **scale spring one-shot** al presionar/soltar (`onPressIn`/`onPressOut`), `useNativeDriver:false`; si estГЎ `disabled`, Pressable no dispara los eventos вҶ’ no anima. **Agrupado + separadores de fecha**: la lista no renderiza `ChatMessageVM[]` directo sino `ChatRow[]` que arma `buildChatRows(messages, now?)` (mГіdulo **puro y testeado** en `view-models/chatRows.ts`): inserta filas `kind:'date'` ("Hoy"/"Ayer"/"d/m/aaaa") al cambiar de dГӯa y marca `groupStart`/`groupEnd` para mensajes consecutivos del mismo remitente (margen amplio al empezar tanda, ajustado entre consecutivas; la hora y "Copiar" se muestran solo en `groupEnd`). El VM expone `ChatMessageVM.createdAtMs` para esto. El `keyExtractor` usa `row.id` estable (`m-<Гӯndice>` / `d-<dГӯaKey>`) вҶ’ no re-anima al apendear. El `HistoryDrawer` usa `expo-blur` (`BlurView`) para el vidrio esmerilado. `Message` ahora tiene `createdAt` (default "ahora"; se rehidrata desde el DTO de persistencia, que lo guarda como ISO opcional para compatibilidad con datos viejos). El input se **ilumina al enfocar** (`onFocus`/`onBlur` вҶ’ borde `primaryBright` + glow) y hay un **botГіn flotante вҶ“** ("bajar al Гәltimo mensaje") que aparece cuando el usuario scrollea hacia arriba (umbral de 140px sobre el fondo, calculado en `onScroll` con `contentSize`/`layoutMeasurement`). **Micro-feedbacks**: el FAB вҶ“ entra con **`PopIn`** (wrapper de entrada one-shot: fade + scale con leve overshoot; su estilo de posiciГіn `scrollDownWrap` va aparte del cГӯrculo `scrollDown`) y usa `PressableScale` al tocar; el botГіn **"Copiar"** tambiГ©n usa `PressableScale` y cambia de color al confirmar (`copyDone` вҶ’ `primaryBright`, ademГЎs del texto "Copiado вң“"). `PopIn` es reutilizable para cualquier elemento que aparece/desaparece вҖ” no es un loop siempre-activo.
- **Adjuntar archivos (рҹ“Һ)**: botГіn clip en el `inputRow` que abre `expo-document-picker` (`getDocumentAsync`). El archivo elegido se muestra como **chip** sobre el input (con вң• para quitarlo). Como los agentes son **solo-texto**, los archivos de texto (`text/*`, `.md`/`.json`/`.csv`/cГіdigoвҖҰ o `mimeType application/json`) se **leen vГӯa `fetch(uri).text()`** y su contenido se inyecta al prompt (truncado a 8000 chars); los binarios (imagen, pdf) van **solo por nombre** como referencia. La combinaciГіn texto+adjunto es lГіgica **pura testeada**: `composeAttachmentMessage(text, attachment)` en `adapters/ui/view-models/` (Vitest), la UI solo elige el archivo. El `submit` compone el mensaje, limpia el chip y lo manda por `send()` вҖ” sin tocar el dominio (un adjunto es solo un prompt enriquecido).
- **VisiГіn (imГЎgenes multimodales)**: el agente **realmente ve** las fotos (no solo el nombre). El dominio `Message` lleva un `imageUrl?` opcional (un **data URL** base64) вҖ” la validaciГіn de texto NO cambia (si solo se manda foto, la UI pone el prompt por defecto `"ВҝQuГ© hay en esta imagen?"`). `SendAssistantQuery.execute(conversation, text, imageUrl?)` lo propaga; el `HuggingFaceAgentAdapter` arma `content` multimodal OpenAI-compatible (`[{type:'text'},{type:'image_url',image_url:{url}}]`) cuando hay imagen, texto plano si no (Ollama lo ignora вҶ’ manda solo texto). La imagen subida por el **usuario (vision)** **NO se persiste** (evita inflar AsyncStorage): es transitoria, solo para el envГӯo y el thumbnail de sesiГіn. (La imagen **generada por el asistente** вҖ”FLUXвҖ” **sГӯ** se persiste; ver bullet "GeneraciГіn de imГЎgenes / persistencia selectiva".) La UI captura la foto como data URL (cГЎmara con `base64:true`; adjuntos `image/*` vГӯa `fileUriToDataUrl` = `fetch(uri).blob()` + `FileReader.readAsDataURL`) y la guarda en `Attachment.previewUri`; `submit` detecta `previewUri` y manda `send(prompt, imageUrl)`. Cubierto por tests en domain/application/adapter/view-model.
- **GeneraciГіn de imГЎgenes (FLUX "Nano Banana") + persistencia selectiva**: el agente `flux` (`HuggingFaceImageAdapter`) implementa `AssistantAgentPort` вҖ” toma el Гәltimo mensaje del usuario como prompt, llama a la API textoвҶ’imagen y devuelve un `Message` de rol `assistant` con `text: 'рҹ“· Imagen generada'` + `imageUrl` (data URL). La burbuja renderiza `imageUrl` de forma **role-agnГіstica** (misma vista para foto subida y para imagen generada), con lightbox. **Persistencia selectiva por rol** en `AsyncStorageConversationRepo`: el DTO de mensaje incluye `imageUrl?` pero al guardar solo se escribe si `message.role === 'assistant'` (la imagen generada es el entregable y debe sobrevivir al reload); la imagen de **vision** (rol `user`) se descarta para no inflar AsyncStorage con base64. Reglas cubiertas: round-trip de la imagen del asistente en el **contrato compartido** (lo cumplen InMemory y AsyncStorage), y el descarte de la imagen de usuario en un spec **especГӯfico de AsyncStorage** (`tests/adapters/AsyncStorageConversationRepo.spec.ts`) вҖ” no va al contrato porque InMemory guarda por referencia y conserva todo. El texto de espera de la burbuja "pensando" es **segГәn la categorГӯa del agente activo**: "рҹҺЁ Generando imagenвҖҰ" para imagen, "рҹҺ¬ Generando videoвҖҰ" para video, "La IA estГЎ pensandoвҖҰ" para chat (la generaciГіn tarda mucho mГЎs que el chat; avisarlo evita que parezca colgado).
- **Sugerencias y label reactivos al agente**: `ChatView` mantiene el proveedor activo en **estado local** (`activeProvider`, init `agentSelector.selected`) y computa `activeCategory = metaFor(activeProvider).category`. El `AgentSelector` recibe una prop **`onChange(provider)`** (la llama en `choose()`) вҶ’ `ChatView` hace `setActiveProvider` y **reacciona en vivo** al cambiar de agente sin recargar: las sugerencias del empty state usan `suggestionsFor(activeCategory)` (mГіdulo puro `view-models/agentSuggestions.ts`: prompts de imagen para FLUX, de video para Krea/Hunyuan, de chat para Ollama/HF вҖ” evita proponer "armame un plan de viaje" con un agente de imГЎgenes) y el texto "pensando" usa `generatingLabel(activeCategory)`. Antes ambos leГӯan `agentSelector.selected` en render, que **no** re-renderiza `ChatView` al cambiar el agente (el `AgentSelector` maneja su propio estado); el `onChange` cierra esa brecha. `ChatEmptyState` recibe `suggestions` por prop (default: chat).
- **Miniatura de imagen en el chip y la burbuja**: el `Attachment` lleva un `previewUri?` (data URL) que el chip renderiza como `<Image>` (38Г—38, Гӯcono рҹ–јпёҸ); `composeAttachmentMessage` lo **ignora** (no va al prompt de texto вҖ” hay un test que lo fija; la imagen viaja por `imageUrl`, no por el texto). La burbuja del usuario tambiГ©n muestra la imagen (`ChatMessageVM.imageUrl` вҶ’ `<Image>` 200Г—150) cuando el mensaje la lleva. Tanto la miniatura del chip como la imagen de la burbuja son tappeables вҶ’ abren un **lightbox** (RN `Modal` transparente, estado `lightboxUri`) a pantalla completa (`resizeMode:'contain'`); tocar cierra.
- **Layout del `inputRow`**: cuadro de texto **centrado** (`flex:1`), con los botones repartidos: izquierda **рҹ“Һ рҹ“·** (adjuntar archivo / foto), derecha **рҹҺӨ + Enviar** (dictar por voz / mandar). Los botones de Гӯcono comparten el estilo `clip` (42Г—46).
- **CГЎmara (рҹ“·)**: botГіn en el `inputRow` que abre la cГЎmara nativa con `expo-image-picker` (`launchCameraAsync`, pide permiso con `requestCameraPermissionsAsync`; el texto del permiso va en el config plugin `expo-image-picker` de `app.json`). La foto se adjunta **reutilizando el mismo chip de adjuntos** y se manda como imagen al modelo de **visiГіn** (ver bullet "VisiГіn"): `launchCameraAsync({ quality: 0.5, base64: true })` вҶ’ data URL en `Attachment.previewUri`. El error de permiso comparte el estado `mediaError` con el micrГіfono.
- **Notas de voz (рҹҺӨ) + transcripciГіn**: botГіn micrГіfono en el `inputRow` (`expo-audio`, `useAudioRecorder(RecordingPresets.HIGH_QUALITY)`). Primer toque graba (pide permiso de micrГіfono vГӯa `AudioModule.requestRecordingPermissionsAsync()`; el mensaje iOS/Android se setea en el config plugin `expo-audio` de `app.json`), segundo toque (вҸ№) para y **transcribe**. Como los agentes son **solo-texto**, el audio NO se manda crudo: se transcribe a texto con **Whisper en HuggingFace** y, al parar, el mensaje se **manda directamente** (si habГӯa texto escrito, lo antepone a lo dictado vГӯa `submit`). El optimista de `ChatViewModel` hace que la burbuja aparezca al instante. El flujo respeta Clean Architecture: nuevo **Port** `TranscriptionPort` (`application/ports`, recibe un `AudioClip { uri, mimeType }`), adapter `HuggingFaceTranscriptionAdapter` (`adapters/ai-agents`), use-case `TranscribeAudio`, y **contrato** `tests/contracts/TranscriptionPort.contract.ts`. **Transporte del binario**: el adapter NO usa `fetch` directo (en RN subir `ArrayBuffer` como body es frГЎgil y tiraba "No se pudo contactar al servicio"); delega en un **`BinaryUpload` inyectable** (`adapters/ai-agents/http.ts`) вҖ” la impl de runtime vive en `App.tsx` y hace `fetch(fileUri).blob()` вҶ’ `POST` con body `Blob` (vГӯa confiable para `file://` en RN), y se **inyecta en `createContainer(env, storage, binaryUpload)`** (como el storage) para que el adapter sea testeable en Node con un fake. El adapter parsea el cuerpo con Zod (`{ text }`) y traduce fallos a `TranscriptionUnavailableError`; el mensaje de error **incluye status + un snippet del cuerpo** para diagnosticar (ej. modelo cargando / no disponible en HF). La UI (`ChatView`) solo graba, infiere el mime de la extensiГіn y pasa `{ uri, mimeType }` al use-case (vГӯa `useDependencies().transcribeAudio`). Estados visibles: "рҹ”ҙ GrabandoвҖҰ" y "TranscribiendoвҖҰ". El STT es **independiente del proveedor de chat** (siempre HuggingFace): vars `EXPO_PUBLIC_AI_STT_BASE_URL` / `_AI_STT_MODEL` / `_AI_STT_API_KEY` (esta Гәltima cae al token del agente si no se setea). Sin key, el adapter falla en runtime y la UI lo muestra sin crashear.
- **UI testeable sin render**: la lГіgica estГЎ en los view-models (`ChatViewModel`, `ConversationListViewModel`) вҖ” TS puro, Vitest. Los hooks (`useAssistant`, `useConversationList`, `useChatViewModel`) solo los conectan a React (`useSyncExternalStore`). Los `.tsx` se typecheckean y lintean, pero no se renderizan en tests ("Vitest para todo").
- **NavegaciГіn**: `@react-navigation/native-stack` con **una sola pantalla** (`Chat`); el historial vive en un **menГә lateral derecho translГәcido** (`HistoryDrawer`, overlay propio sin `@react-navigation/drawer` para no sumar `reanimated`/`gesture-handler`) que se abre con el botГіn вҳ° del header. Seleccionar/crear una conversaciГіn hace `navigation.setParams({ conversationId })` вҶ’ el hook `useChatViewModel` reconstruye el view-model. Params serializables (`conversationId`, no instancias). El drawer desliza con `Animated.timing` (one-shot, sin loop de rAF). `App.tsx` monta `SafeAreaProvider вҶ’ DependenciesProvider вҶ’ NavigationContainer вҶ’ RootNavigator`.
- **Selector rГЎpido de conversaciones (multi-conversaciГіn Fase A)**: tira horizontal de chips arriba del chat (`ConversationTabs`, debajo del `AgentSelector` en `ChatView`) para saltar entre charlas **sin abrir el drawer**, con la activa resaltada y un chip **пјӢ** para crear una nueva. Los chips los arma un **mГіdulo puro y testeado** `view-models/conversationTabs.ts` (`buildConversationTabs(items, activeId)` вҶ’ `{ id, label, active }[]`; label prefiere `title` sobre `preview`): si la conversaciГіn activa es **nueva y aГәn no estГЎ persistida** (no aparece en `items` hasta el primer mensaje) le agrega un chip sintГ©tico "Nueva charla" activo. El componente reusa `useConversationList` (recarga al montar y al cambiar `activeId`) y delega el cambio en los mismos callbacks del drawer: `ChatScreen` pasa `activeConversationId` + `onSelectConversation`/`onNewConversation` вҶ’ `ChatView` вҶ’ `ConversationTabs`; tap en el chip activo es no-op. Combina con la Fase B (abajo): los `ChatViewModel` quedan vivos en un registry, asГӯ al volver a una charla se ve su estado en vivo (incluido un envГӯo en vuelo), no solo el estado persistido.
- **Concurrencia en vivo (multi-conversaciГіn Fase B)**: `ChatViewModelRegistry` (`adapters/ui/registry/`) mantiene **vivas** las instancias de `ChatViewModel` por `conversationId` en un `Map`. Antes `useChatViewModel` reconstruГӯa el VM en cada cambio de charla y descartaba el anterior, perdiendo el estado en vivo (el indicador "pensandoвҖҰ", un envГӯo en vuelo); ahora **reutiliza la misma instancia**, asГӯ un envГӯo disparado en la charla A sigue actualizando su VM aunque estГ©s viendo la B, y al volver a A ves su estado real (en vuelo o ya respondido). `getOrCreate(id)` carga del repo vГӯa `GetConversation` (o `Conversation.start` si no existe) y cachea, con guard de concurrencia (`pending` map) para no duplicar instancias ante llamadas simultГЎneas; `peek(id)` es **sincrГіnico** y lo usa `useChatViewModel` para setear el estado inicial sin el flash de "cargando" al volver a una charla ya abierta; `evict(id)` descarta el VM (lo llama el `HistoryDrawer` al borrar la conversaciГіn). El registry se instancia **una vez** en `App.tsx` (`useMemo` sobre el container; se recrea solo al reiniciar tras guardar ajustes) y se inyecta vГӯa `UiDependencies.chatViewModelRegistry`. Para el caso sin `conversationId` (primer arranque sin params) el hook usa un id estable por `useRef` para que esa charla tambiГ©n quede cacheada. Cubierto por tests del registry (misma instancia por id, siembra desde el repo, concurrencia, evict).
- **Exportar / compartir conversaciГіn**: botГіn **рҹ“Ө** en el `headerRight` del chat (junto al вҳ°) que comparte la conversaciГіn activa como **texto plano** vГӯa el **`Share` nativo de React Native** (`Share.share({ message })`) вҖ” NO se usГі `expo-sharing` (comparte archivos, no texto, y sumarГӯa una dep nativa + `expo-file-system` para escribir el .txt). El texto lo arma un **mГіdulo puro y testeado** `view-models/formatConversationAsText.ts` (`Conversation` вҶ’ string: encabezado con `title ?? 'ConversaciГіn'` + turnos etiquetados `TГә:` / `Asistente:`; **sin fechas** para que sea determinГӯstico/testeable). El `ChatViewModel` expone `exportText()` (delega en el formatter sobre su `conversation`) y `hasMessages`. El botГіn se **habilita reactivamente**: `ChatScreen` se suscribe al VM con un `useSyncExternalStore` propio que tolera `viewModel === null` (`messages.length > 0` вҶ’ `canShare`, en deps del `useLayoutEffect`) para que se prenda en cuanto la charla tiene mensajes, sin esperar a cambiar de conversaciГіn. El `Share` falla/cancela sin romper (try/catch); en web `Share` puede no estar soportado (acciГіn nativa).
- **GestiГіn de conversaciones (drawer)**: cada fila tiene **Renombrar** y **Borrar**. Borrar usa `DeleteConversation` (ya existГӯa); **renombrar** es un slice vertical: `Conversation` ganГі un `title?` opcional + `rename(t)` (valida no vacГӯo, se permite aГәn cerrada) вҖ” la `start(id, title?)` lo acepta. Se **persiste** (DTO de AsyncStorage con `title?` opcional). Nuevo use-case `RenameConversation` (findвҶ’renameвҶ’save; no-op si el id no existe o el tГӯtulo viene vacГӯo) cableado en `container` + `UiDependencies` + `useConversationList` (3er arg del `ConversationListViewModel`). El `ConversationSummaryVM` expone `title`; la fila muestra `title ?? preview`. La UI es un **modal propio con `TextInput`** (no `Alert.prompt`, que es solo iOS) вҖ” `viewModel.rename(id, texto)` recarga la lista. Cubierto por tests en domain/contract(persistencia)/application/view-model.
- **BГәsqueda en el historial (drawer)**: barra de bГәsqueda (`TextInput` con рҹ”Қ y вң• para limpiar) sobre la lista. El filtrado es un **mГіdulo puro y testeado** `view-models/filterConversations.ts` (`filterConversations(items, query)`): match **case- y acento-insensible** (`normalize` = `NFD` + quita `U+0300вҖ“U+036F` + lowercase) contra `title` + `preview` + `searchText`; query vacГӯa o con solo espacios вҶ’ devuelve todos. Para que la bГәsqueda encuentre una palabra que solo aparece **en medio** de la charla (no solo en tГӯtulo/preview), `toSummary` arma `ConversationSummaryVM.searchText` = tГӯtulo + **todos** los textos de mensajes concatenados (campo oculto, no se muestra). El drawer mantiene el `query` en estado local (se limpia al cerrarse) y la `FlatList` renderiza `filterConversations(state.items, query)`; el `ListEmptyComponent` distingue "Sin conversaciones" de `Sin resultados para "вҖҰ"`. La UI no toca el `ConversationListViewModel` (el filtro corre sobre los summaries ya cargados).
- **DI en la UI sin romper capas**: la UI NO importa `infrastructure`. El container se inyecta vГӯa React Context (`DependenciesProvider` / `useDependencies` en `adapters/ui/di/`), tipado como `UiDependencies` (use-cases de `application` + `AgentSelector`). El `Container` concreto satisface esa interfaz estructuralmente al pasarse al Provider.
- **Tema visual**: paleta, tipografГӯa, espaciado, radios y un helper `glow()` centralizados en `adapters/ui/theme/theme.ts` вҖ” los componentes referencian tokens, no hex sueltos. La display (`Space Grotesk`) se carga async con `useFonts` en `App.tsx` (que gatea el render hasta que estГ© lista); `font.display`/`font.displayBold` apuntan a esos nombres. El `AgentSelector` muestra el modelo activo (badge) y un punto de estado **estГЎtico** con glow (sin animaciГіn: un loop de Animated rompe render/perf en web). Fondo con gradiente (`expo-linear-gradient`) + motivo IA (`AiBackdrop`, orbes con glow hechos con `View`s). El componente **`ScreenBackground`** (gradiente + `AiBackdrop` como capa `absoluteFill` sin capturar toques) unifica el fondo entre pantallas: lo usan **Onboarding** y **Ajustes** (montado como primer hijo del contenedor, el contenido se pinta encima). El **Chat** mantiene su propia versiГіn porque le suma parallax de scroll. Parallax con `onScroll` simple + estado (NO `Animated.event`/`Animated.FlatList`: en react-native-web instalan un loop de `requestAnimationFrame` continuo que satura el render y rompe capturas/perf). Texto claro sobre fondo oscuro para contraste. El selector de agente son chips (`AgentSelector`) que llaman a `agentSelector.select()`.
