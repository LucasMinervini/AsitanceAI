## Proyecto

App Mobile de Asistencia IA que integra agentes de IA gratuitos. TypeScript + pnpm, diseñada con Clean Architecture y desarrollada con TDD (Spec-First). El núcleo (`domain` + `application` + adapters) se construyó y probó **sin framework** hasta tener cobertura verde; recién entonces se instaló **React Native / Expo** (Expo SDK 54, RN 0.81, React 19.1) como capa de UI. El núcleo sigue siendo independiente del framework: la lógica vive en view-models de TS puro testeables con Vitest, y RN solo renderiza.

## Comandos

Gestor de paquetes: **pnpm** (vía corepack, versión fijada en `packageManager`). No usar npm/yarn — `.npmrc` impone `node-linker=isolated` para evitar dependencias fantasma.

```bash
pnpm install              # instalar dependencias
pnpm start                # expo start (dev server / Metro en http://localhost:8081)
pnpm tunnel               # expo start --tunnel (probar en celular fuera de la LAN; usa @expo/ngrok)
pnpm web                  # expo start --web (bundle web, sin emulador)
pnpm android / pnpm ios   # abrir en emulador/simulador
pnpm test                 # vitest run (toda la suite, una vez)
pnpm test:watch           # modo watch para el ciclo RED→GREEN→REFACTOR
pnpm test:domain          # solo tests/domain
pnpm test:application     # solo tests/application
pnpm test:contracts       # solo tests/contracts
pnpm test:infrastructure  # solo tests/infrastructure
pnpm test:coverage        # con cobertura v8 (umbral 80% en domain/application)
pnpm typecheck            # tsc --noEmit (forma parte del "verde": debe pasar siempre)
pnpm lint                 # eslint, --max-warnings=0 (hace cumplir la frontera de capas)
pnpm build                # tsc -p tsconfig.build.json → dist/ (excluye tests y dominio puro de tipos)
pnpm format               # prettier --write
```

Ejecutar **un solo archivo o un test concreto**:

```bash
pnpm exec vitest run tests/domain/Conversation.spec.ts
pnpm exec vitest run -t "no permite agregar mensajes a una conversacion cerrada"
```

## Arquitectura

Capas con la **regla de dependencia hacia adentro** (`infrastructure → adapters → application → domain`). El sentido de los imports lo hace cumplir ESLint (`no-restricted-imports` en `eslint.config.js`): `src/domain/**` no puede importar de ninguna otra capa, y `src/application/**` no puede importar de `adapters` ni `infrastructure`. Si un import cruza una frontera prohibida, `pnpm lint` falla — no es solo convención.

- **`src/domain/`** — Entities y Value Objects. Lógica pura, sin I/O ni dependencias externas. Las entidades usan **constructor privado + factory estática** (`Message.create()`, `Conversation.start()`), son inmutables hacia afuera, se autovalidan en construcción y devuelven copias defensivas (ej. `Conversation.history`). Las invariantes de negocio se rompen lanzando un error que extiende `DomainError` (ej. `ConversationClosedError`).
- **`src/application/`** — Use Cases y **Ports** (interfaces). El use-case recibe los Ports por constructor (DIP) y orquesta el flujo. Los Ports expresan el *qué* necesita la app (ej. `AssistantAgentPort`), nunca el *cómo*.
- **`src/adapters/`** — Implementaciones concretas: `ai-agents/` (`OllamaLocalAgentAdapter` local sin auth y `HuggingFaceAgentAdapter` con Bearer token; comparten el `FetchLike` inyectable de `ai-agents/http.ts`; cada uno mapea su DTO crudo —forma distinta— al dominio con Zod y traduce fallos a `AgentUnavailableError`), `persistence/` (`InMemoryConversationRepo` para tests + `AsyncStorageConversationRepo` real, que serializa a un DTO Zod y depende de una interfaz `KeyValueStorage` inyectable, no de AsyncStorage directo), `ui/`. `di/createAssistantAgents.ts` construye un agente por proveedor (record exhaustivo sobre `AiAgentProvider`) y `RoutingAssistantAgent` (adapter genérico) enruta a uno según el proveedor seleccionado, cambiable **en runtime** vía `select()`. El proveedor inicial es `AI_AGENT_PROVIDER`. El container expone un `AgentSelector` a la UI; el contrato garantiza que todos los agentes se comportan igual. La UI usa **view-models framework-agnostic** (`ChatViewModel`): un presenter en TS puro con patrón store (`getState()` + `subscribe()`) que mapea las entidades a DTOs de presentación. Un hook de React/RN lo envolverá con `useSyncExternalStore` cuando se instale Expo — la lógica de UI es testeable con Vitest sin renderizar.
- **`src/infrastructure/`** — Composition Root: único lugar donde se instancian clases concretas y se inyectan (`di/container.ts`). `config/env.ts` es el **único** sitio que lee `process.env` y valida con Zod.
- **`src/shared/`** — Transversal. `Result<T,E>` (`result/Result.ts`) para errores explícitos sin `throw` en application/adapters, y la jerarquía de errores tipados (`errors/`).

### Convención de errores

Dos mecanismos coexisten a propósito: el **dominio lanza** (`throw`) subclases de `DomainError` para violar invariantes; **application y adapters devuelven `Result<T, E>`** en lugar de lanzar, de modo que el camino de error queda en la firma del tipo.

## Tests (TDD)

`tests/` está separado por nivel, espejando las capas:
- `tests/domain/` — unitarios puros.
- `tests/application/` — aceptación de use-cases con **Fakes/Stubs** de los Ports (sin I/O real).
- `tests/contracts/` — un mismo contrato (`*.contract.ts`) corre contra cada adaptador de un Port para garantizar que todos lo cumplen igual (IA y persistencia). El archivo `*.contract.ts` es autocontenido: define la función de contrato y la invoca con la factory de cada adaptador al final del archivo (debe contener `describe`, no solo exportar).
- `tests/builders/` — **Test Data Builders** (`MessageBuilder.aMessage().withText(...).build()`). Crear datos de prueba siempre vía Builders, no construyendo entidades a mano.
- `tests/fakes/` — **Fakes reutilizables** de los Ports (ej. `FakeAssistantAgent`), con factories por escenario (`thatReplies(...)`, `thatIsUnavailable()`).
- `tests/infrastructure/` — Composition Root. `loadEnv()` se prueba con una fuente de env inyectada (no `process.env` real); `createContainer()` verifica que el grafo queda cableado.
- `tests/adapters/` — adaptadores sin contrato de Port, ej. los view-models de UI (`ChatViewModel`) probados como lógica pura con Vitest.

Glob de Vitest: `tests/**/*.{spec,test,contract}.ts`. Los alias de import (`@domain/*`, `@application/*`, `@adapters/*`, `@infrastructure/*`, `@shared/*`) se definen una sola vez en `tsconfig.json` y Vitest los lee vía `vite-tsconfig-paths` — no duplicar alias en `vitest.config.ts`.

Flujo: escribir el test que falla (RED) → mínimo código para pasar (GREEN) → refactor, subiendo de capa hacia afuera. `pnpm typecheck` cuenta como parte del verde.

## Expo / React Native

El toolchain de Expo (Metro/Babel) impuso decisiones que conviene conocer:

- **SDK fijado en 54 por Expo Go (iOS)**: en iPhone **físico**, el Expo Go del App Store soporta como máximo **SDK 54** (SDK 55/56 requieren `eas go`/TestFlight con cuenta Apple paga). Por eso el proyecto está en SDK 54 aunque exista 56. Si se sube de SDK, deja de abrir en Expo Go estándar.
- **`expo-status-bar` NO es config plugin**: es un componente que se importa, no va en `app.json > plugins`. Ponerlo ahí crashea el arranque en SDK 54 (`Unable to resolve a valid config plugin for expo-status-bar`). En `plugins` solo van plugins reales (ej. `expo-font`).
- **`package.json` NO tiene `"type": "module"`** — los configs de Expo (`babel.config.js`, `metro.config.js`) son CommonJS. Por eso el flat config de ESLint es `eslint.config.mjs` y el de Vitest es `vitest.config.mts` (necesitan cargarse como ESM).
- **pnpm `isolated` + build tooling**: el ecosistema RN asume `node_modules` aplanado. Mantenemos `node-linker=isolated` a propósito; cuando Metro/Babel falle con `MODULE_NOT_FOUND` por un preset/plugin (ej. `babel-preset-expo`), la solución es **declararlo como devDependency explícita** (es justo lo que pide el principio anti-phantom), NO cambiar a `hoisted`.
- **Alias en Metro**: Expo lee los `paths` de `tsconfig.json` automáticamente — no duplicar alias en `metro.config.js`.
- **Entry point**: `index.js` → `registerRootComponent(App)`. `src/infrastructure/app/App.tsx` es el Composition Root del runtime: `createContainer(loadEnv(expoEnvSource()), AsyncStorage)` y arma el `ChatViewModel`. `createContainer` recibe el `KeyValueStorage` por parámetro: el runtime pasa AsyncStorage, los tests un fake — así el container es testeable en Node sin el módulo nativo.
- **Caché de Metro en Windows/OneDrive**: el crawler incremental puede fallar con `EINVAL: readlink` sobre symlinks de pnpm al aplicar el delta de caché. Solución: `expo start --clear` / `expo export --clear` (fuerza crawl limpio; el bundle funciona bien).
- **Env en RN**: no hay `.env` en runtime; `expoEnvSource()` (en `config/env.ts`) lee las vars `EXPO_PUBLIC_*` que Babel inyecta en build, con defaults locales. `process.env` sigue confinado a `config/env.ts`. `loadEnv` valida con Zod, incluido un `superRefine` que exige `AI_AGENT_API_KEY` cuando `AI_AGENT_PROVIDER=huggingface`. También expone la config de transcripción (`AI_STT_BASE_URL` / `AI_STT_MODEL` y `AI_STT_API_KEY` opcional que en `expoEnvSource()` cae al token del agente). **El default de `AI_STT_BASE_URL` es `https://router.huggingface.co/hf-inference/models`** (NO el viejo `api-inference.huggingface.co`, que HuggingFace dio de baja: tiraba `Network request failed` aunque el chat —que ya usa `router.huggingface.co`— funcionara). El adapter arma `${baseUrl}/${model}` → `…/hf-inference/models/openai/whisper-large-v3`. `App.tsx` envuelve la creación del container en un `try/catch` y muestra una pantalla "Error de configuración" en vez de crashear si la env es inválida. Esa pantalla incluye un botón **"Borrar ajustes guardados y reintentar"** (red de seguridad anti-brick): borra `settings:apiKey` + `settings:ollamaBaseUrl` de AsyncStorage y reinicia (`bootKey++`) con los defaults del build, sin reinstalar. Es la **única vía de vuelta** si un override guardado hace fallar `loadEnv`. Para que ese caso casi no ocurra, `SettingsViewModel.save()` **valida la URL de Ollama antes de persistir** con una regla más estricta que env (`z.string().url().refine(http(s)://)`): Zod `.url()` por sí solo acepta `localhost:11434` (esquema `localhost:`), que pasaría env pero el adapter no puede usar; URL inválida → `phase:'error'` con mensaje inline y **no toca persistencia** (cubierto por tests).
- **`.env` vs `.env.local` (Expo SDK 56)**: `@expo/env` 2.x **rechaza** variables que no empiezan con `EXPO_PUBLIC_` dentro de `.env` (`Refused to load personal environment variables from a non-.local env file`). Las "personales" (ej. `REACT_NATIVE_PACKAGER_HOSTNAME` para forzar la IP del QR en LAN, secretos) van en **`.env.local`**; en `.env` solo `EXPO_PUBLIC_*`. Ambos están gitignoreados.
- **Borrar `dist-web` (export web) en Windows**: las rutas de assets superan MAX_PATH y `Remove-Item` falla. Usar `robocopy <dir-vacío> dist-web /MIR` y luego borrar. `dist-web/**` está en `.gitignore` y en los `ignores` de ESLint.
- **Teclado / detalles de chat**: el chat usa `KeyboardAvoidingView` con `useHeaderHeight()` (de `@react-navigation/elements`) para que el input suba sobre el teclado en iOS. El **safe-area inferior** lo aplica la fila de input vía `useSafeAreaInsets()` → `marginBottom: Math.max(insets.bottom, spacing.lg)` (el `KeyboardAvoidingView` no reserva el home indicator y los botones quedaban cortados con el teclado cerrado); el `content` ya no usa `paddingBottom` (lo maneja la fila) y el FAB ↓ calcula su `bottom` con el mismo inset. Las burbujas muestran hora (`Message.createdAt` → `ChatMessageVM.time`) y botón "Copiar" (`expo-clipboard`); auto-scroll al último mensaje. **Envío con UI optimista**: `ChatViewModel.send()` arma una burbuja provisoria del usuario y la muestra **al instante** (status `sending` + "La IA está pensando…") en vez de esperar el round-trip; al terminar, el snapshot real del `conversation.history` (user + assistant) la reemplaza. En error, el snapshot conserva la pregunta. Está cubierto por test (el primer cambio a `sending` ya trae el mensaje del usuario). Cada burbuja entra con una animación **one-shot** (fade + slide-up + leve scale 0.96→1, `AnimatedBubble`): corre una vez al montar (keyExtractor estable → no re-anima en cada render), `useNativeDriver:false` para compat web como el drawer. La burbuja "pensando" también entra animada. Los botones del `inputRow` (📎 📷 🎤 Enviar) usan **`PressableScale`** (componente reutilizable): hace un **scale spring one-shot** al presionar/soltar (`onPressIn`/`onPressOut`), `useNativeDriver:false`; si está `disabled`, Pressable no dispara los eventos → no anima. **Agrupado + separadores de fecha**: la lista no renderiza `ChatMessageVM[]` directo sino `ChatRow[]` que arma `buildChatRows(messages, now?)` (módulo **puro y testeado** en `view-models/chatRows.ts`): inserta filas `kind:'date'` ("Hoy"/"Ayer"/"d/m/aaaa") al cambiar de día y marca `groupStart`/`groupEnd` para mensajes consecutivos del mismo remitente (margen amplio al empezar tanda, ajustado entre consecutivas; la hora y "Copiar" se muestran solo en `groupEnd`). El VM expone `ChatMessageVM.createdAtMs` para esto. El `keyExtractor` usa `row.id` estable (`m-<índice>` / `d-<díaKey>`) → no re-anima al apendear. El `HistoryDrawer` usa `expo-blur` (`BlurView`) para el vidrio esmerilado. `Message` ahora tiene `createdAt` (default "ahora"; se rehidrata desde el DTO de persistencia, que lo guarda como ISO opcional para compatibilidad con datos viejos). El input se **ilumina al enfocar** (`onFocus`/`onBlur` → borde `primaryBright` + glow) y hay un **botón flotante ↓** ("bajar al último mensaje") que aparece cuando el usuario scrollea hacia arriba (umbral de 140px sobre el fondo, calculado en `onScroll` con `contentSize`/`layoutMeasurement`).
- **Adjuntar archivos (📎)**: botón clip en el `inputRow` que abre `expo-document-picker` (`getDocumentAsync`). El archivo elegido se muestra como **chip** sobre el input (con ✕ para quitarlo). Como los agentes son **solo-texto**, los archivos de texto (`text/*`, `.md`/`.json`/`.csv`/código… o `mimeType application/json`) se **leen vía `fetch(uri).text()`** y su contenido se inyecta al prompt (truncado a 8000 chars); los binarios (imagen, pdf) van **solo por nombre** como referencia. La combinación texto+adjunto es lógica **pura testeada**: `composeAttachmentMessage(text, attachment)` en `adapters/ui/view-models/` (Vitest), la UI solo elige el archivo. El `submit` compone el mensaje, limpia el chip y lo manda por `send()` — sin tocar el dominio (un adjunto es solo un prompt enriquecido).
- **Visión (imágenes multimodales)**: el agente **realmente ve** las fotos (no solo el nombre). El dominio `Message` lleva un `imageUrl?` opcional (un **data URL** base64) — la validación de texto NO cambia (si solo se manda foto, la UI pone el prompt por defecto `"¿Qué hay en esta imagen?"`). `SendAssistantQuery.execute(conversation, text, imageUrl?)` lo propaga; el `HuggingFaceAgentAdapter` arma `content` multimodal OpenAI-compatible (`[{type:'text'},{type:'image_url',image_url:{url}}]`) cuando hay imagen, texto plano si no (Ollama lo ignora → manda solo texto). La imagen subida por el **usuario (vision)** **NO se persiste** (evita inflar AsyncStorage): es transitoria, solo para el envío y el thumbnail de sesión. (La imagen **generada por el asistente** —FLUX— **sí** se persiste; ver bullet "Generación de imágenes / persistencia selectiva".) La UI captura la foto como data URL (cámara con `base64:true`; adjuntos `image/*` vía `fileUriToDataUrl` = `fetch(uri).blob()` + `FileReader.readAsDataURL`) y la guarda en `Attachment.previewUri`; `submit` detecta `previewUri` y manda `send(prompt, imageUrl)`. Cubierto por tests en domain/application/adapter/view-model.
- **Generación de imágenes (FLUX "Nano Banana") + persistencia selectiva**: el agente `flux` (`HuggingFaceImageAdapter`) implementa `AssistantAgentPort` — toma el último mensaje del usuario como prompt, llama a la API texto→imagen y devuelve un `Message` de rol `assistant` con `text: '📷 Imagen generada'` + `imageUrl` (data URL). La burbuja renderiza `imageUrl` de forma **role-agnóstica** (misma vista para foto subida y para imagen generada), con lightbox. **Persistencia selectiva por rol** en `AsyncStorageConversationRepo`: el DTO de mensaje incluye `imageUrl?` pero al guardar solo se escribe si `message.role === 'assistant'` (la imagen generada es el entregable y debe sobrevivir al reload); la imagen de **vision** (rol `user`) se descarta para no inflar AsyncStorage con base64. Reglas cubiertas: round-trip de la imagen del asistente en el **contrato compartido** (lo cumplen InMemory y AsyncStorage), y el descarte de la imagen de usuario en un spec **específico de AsyncStorage** (`tests/adapters/AsyncStorageConversationRepo.spec.ts`) — no va al contrato porque InMemory guarda por referencia y conserva todo. El texto de espera de la burbuja "pensando" es **según la categoría del agente activo** (`generatingLabel(metaFor(agentSelector.selected).category)` en `ChatView`): "🎨 Generando imagen…" para imagen, "🎬 Generando video…" para video, "La IA está pensando…" para chat (la generación tarda mucho más que el chat; avisarlo evita que parezca colgado). Se lee en render; al enviar, el provider ya quedó fijado por el selector.
- **Miniatura de imagen en el chip y la burbuja**: el `Attachment` lleva un `previewUri?` (data URL) que el chip renderiza como `<Image>` (38×38, ícono 🖼️); `composeAttachmentMessage` lo **ignora** (no va al prompt de texto — hay un test que lo fija; la imagen viaja por `imageUrl`, no por el texto). La burbuja del usuario también muestra la imagen (`ChatMessageVM.imageUrl` → `<Image>` 200×150) cuando el mensaje la lleva. Tanto la miniatura del chip como la imagen de la burbuja son tappeables → abren un **lightbox** (RN `Modal` transparente, estado `lightboxUri`) a pantalla completa (`resizeMode:'contain'`); tocar cierra.
- **Layout del `inputRow`**: cuadro de texto **centrado** (`flex:1`), con los botones repartidos: izquierda **📎 📷** (adjuntar archivo / foto), derecha **🎤 + Enviar** (dictar por voz / mandar). Los botones de ícono comparten el estilo `clip` (42×46).
- **Cámara (📷)**: botón en el `inputRow` que abre la cámara nativa con `expo-image-picker` (`launchCameraAsync`, pide permiso con `requestCameraPermissionsAsync`; el texto del permiso va en el config plugin `expo-image-picker` de `app.json`). La foto se adjunta **reutilizando el mismo chip de adjuntos** y se manda como imagen al modelo de **visión** (ver bullet "Visión"): `launchCameraAsync({ quality: 0.5, base64: true })` → data URL en `Attachment.previewUri`. El error de permiso comparte el estado `mediaError` con el micrófono.
- **Notas de voz (🎤) + transcripción**: botón micrófono en el `inputRow` (`expo-audio`, `useAudioRecorder(RecordingPresets.HIGH_QUALITY)`). Primer toque graba (pide permiso de micrófono vía `AudioModule.requestRecordingPermissionsAsync()`; el mensaje iOS/Android se setea en el config plugin `expo-audio` de `app.json`), segundo toque (⏹) para y **transcribe**. Como los agentes son **solo-texto**, el audio NO se manda crudo: se transcribe a texto con **Whisper en HuggingFace** y, al parar, el mensaje se **manda directamente** (si había texto escrito, lo antepone a lo dictado vía `submit`). El optimista de `ChatViewModel` hace que la burbuja aparezca al instante. El flujo respeta Clean Architecture: nuevo **Port** `TranscriptionPort` (`application/ports`, recibe un `AudioClip { uri, mimeType }`), adapter `HuggingFaceTranscriptionAdapter` (`adapters/ai-agents`), use-case `TranscribeAudio`, y **contrato** `tests/contracts/TranscriptionPort.contract.ts`. **Transporte del binario**: el adapter NO usa `fetch` directo (en RN subir `ArrayBuffer` como body es frágil y tiraba "No se pudo contactar al servicio"); delega en un **`BinaryUpload` inyectable** (`adapters/ai-agents/http.ts`) — la impl de runtime vive en `App.tsx` y hace `fetch(fileUri).blob()` → `POST` con body `Blob` (vía confiable para `file://` en RN), y se **inyecta en `createContainer(env, storage, binaryUpload)`** (como el storage) para que el adapter sea testeable en Node con un fake. El adapter parsea el cuerpo con Zod (`{ text }`) y traduce fallos a `TranscriptionUnavailableError`; el mensaje de error **incluye status + un snippet del cuerpo** para diagnosticar (ej. modelo cargando / no disponible en HF). La UI (`ChatView`) solo graba, infiere el mime de la extensión y pasa `{ uri, mimeType }` al use-case (vía `useDependencies().transcribeAudio`). Estados visibles: "🔴 Grabando…" y "Transcribiendo…". El STT es **independiente del proveedor de chat** (siempre HuggingFace): vars `EXPO_PUBLIC_AI_STT_BASE_URL` / `_AI_STT_MODEL` / `_AI_STT_API_KEY` (esta última cae al token del agente si no se setea). Sin key, el adapter falla en runtime y la UI lo muestra sin crashear.
- **UI testeable sin render**: la lógica está en los view-models (`ChatViewModel`, `ConversationListViewModel`) — TS puro, Vitest. Los hooks (`useAssistant`, `useConversationList`, `useChatViewModel`) solo los conectan a React (`useSyncExternalStore`). Los `.tsx` se typecheckean y lintean, pero no se renderizan en tests ("Vitest para todo").
- **Navegación**: `@react-navigation/native-stack` con **una sola pantalla** (`Chat`); el historial vive en un **menú lateral derecho translúcido** (`HistoryDrawer`, overlay propio sin `@react-navigation/drawer` para no sumar `reanimated`/`gesture-handler`) que se abre con el botón ☰ del header. Seleccionar/crear una conversación hace `navigation.setParams({ conversationId })` → el hook `useChatViewModel` reconstruye el view-model. Params serializables (`conversationId`, no instancias). El drawer desliza con `Animated.timing` (one-shot, sin loop de rAF). `App.tsx` monta `SafeAreaProvider → DependenciesProvider → NavigationContainer → RootNavigator`.
- **Selector rápido de conversaciones (multi-conversación Fase A)**: tira horizontal de chips arriba del chat (`ConversationTabs`, debajo del `AgentSelector` en `ChatView`) para saltar entre charlas **sin abrir el drawer**, con la activa resaltada y un chip **＋** para crear una nueva. Los chips los arma un **módulo puro y testeado** `view-models/conversationTabs.ts` (`buildConversationTabs(items, activeId)` → `{ id, label, active }[]`; label prefiere `title` sobre `preview`): si la conversación activa es **nueva y aún no está persistida** (no aparece en `items` hasta el primer mensaje) le agrega un chip sintético "Nueva charla" activo. El componente reusa `useConversationList` (recarga al montar y al cambiar `activeId`) y delega el cambio en los mismos callbacks del drawer: `ChatScreen` pasa `activeConversationId` + `onSelectConversation`/`onNewConversation` → `ChatView` → `ConversationTabs`; tap en el chip activo es no-op. Como **una sola `ChatViewModel` vive a la vez** (se reconstruye al cambiar el param) pero los resultados **se persisten**, al volver a una charla se ve el estado final (incluida la imagen de FLUX), aunque no el indicador "pensando…" en vivo mientras se está en otra. La concurrencia en vivo (registry de view-models) es **Fase B** pendiente.
- **Exportar / compartir conversación**: botón **📤** en el `headerRight` del chat (junto al ☰) que comparte la conversación activa como **texto plano** vía el **`Share` nativo de React Native** (`Share.share({ message })`) — NO se usó `expo-sharing` (comparte archivos, no texto, y sumaría una dep nativa + `expo-file-system` para escribir el .txt). El texto lo arma un **módulo puro y testeado** `view-models/formatConversationAsText.ts` (`Conversation` → string: encabezado con `title ?? 'Conversación'` + turnos etiquetados `Tú:` / `Asistente:`; **sin fechas** para que sea determinístico/testeable). El `ChatViewModel` expone `exportText()` (delega en el formatter sobre su `conversation`) y `hasMessages`. El botón se **habilita reactivamente**: `ChatScreen` se suscribe al VM con un `useSyncExternalStore` propio que tolera `viewModel === null` (`messages.length > 0` → `canShare`, en deps del `useLayoutEffect`) para que se prenda en cuanto la charla tiene mensajes, sin esperar a cambiar de conversación. El `Share` falla/cancela sin romper (try/catch); en web `Share` puede no estar soportado (acción nativa).
- **Gestión de conversaciones (drawer)**: cada fila tiene **Renombrar** y **Borrar**. Borrar usa `DeleteConversation` (ya existía); **renombrar** es un slice vertical: `Conversation` ganó un `title?` opcional + `rename(t)` (valida no vacío, se permite aún cerrada) — la `start(id, title?)` lo acepta. Se **persiste** (DTO de AsyncStorage con `title?` opcional). Nuevo use-case `RenameConversation` (find→rename→save; no-op si el id no existe o el título viene vacío) cableado en `container` + `UiDependencies` + `useConversationList` (3er arg del `ConversationListViewModel`). El `ConversationSummaryVM` expone `title`; la fila muestra `title ?? preview`. La UI es un **modal propio con `TextInput`** (no `Alert.prompt`, que es solo iOS) — `viewModel.rename(id, texto)` recarga la lista. Cubierto por tests en domain/contract(persistencia)/application/view-model.
- **Búsqueda en el historial (drawer)**: barra de búsqueda (`TextInput` con 🔍 y ✕ para limpiar) sobre la lista. El filtrado es un **módulo puro y testeado** `view-models/filterConversations.ts` (`filterConversations(items, query)`): match **case- y acento-insensible** (`normalize` = `NFD` + quita `U+0300–U+036F` + lowercase) contra `title` + `preview` + `searchText`; query vacía o con solo espacios → devuelve todos. Para que la búsqueda encuentre una palabra que solo aparece **en medio** de la charla (no solo en título/preview), `toSummary` arma `ConversationSummaryVM.searchText` = título + **todos** los textos de mensajes concatenados (campo oculto, no se muestra). El drawer mantiene el `query` en estado local (se limpia al cerrarse) y la `FlatList` renderiza `filterConversations(state.items, query)`; el `ListEmptyComponent` distingue "Sin conversaciones" de `Sin resultados para "…"`. La UI no toca el `ConversationListViewModel` (el filtro corre sobre los summaries ya cargados).
- **DI en la UI sin romper capas**: la UI NO importa `infrastructure`. El container se inyecta vía React Context (`DependenciesProvider` / `useDependencies` en `adapters/ui/di/`), tipado como `UiDependencies` (use-cases de `application` + `AgentSelector`). El `Container` concreto satisface esa interfaz estructuralmente al pasarse al Provider.
- **Tema visual**: paleta, tipografía, espaciado, radios y un helper `glow()` centralizados en `adapters/ui/theme/theme.ts` — los componentes referencian tokens, no hex sueltos. La display (`Space Grotesk`) se carga async con `useFonts` en `App.tsx` (que gatea el render hasta que esté lista); `font.display`/`font.displayBold` apuntan a esos nombres. El `AgentSelector` muestra el modelo activo (badge) y un punto de estado **estático** con glow (sin animación: un loop de Animated rompe render/perf en web). Fondo con gradiente (`expo-linear-gradient`) + motivo IA (`AiBackdrop`, orbes con glow hechos con `View`s). Parallax con `onScroll` simple + estado (NO `Animated.event`/`Animated.FlatList`: en react-native-web instalan un loop de `requestAnimationFrame` continuo que satura el render y rompe capturas/perf). Texto claro sobre fondo oscuro para contraste. El selector de agente son chips (`AgentSelector`) que llaman a `agentSelector.select()`.
- **Evitar `react-native-svg` en web**: su build web (`ReactNativeSVG.web.js`) no resuelve bajo Metro/pnpm (`Unable to resolve ./lib/extract/types`). Por eso el `AiBackdrop` se hizo con `View`s en vez de SVG — funciona en web y nativo sin dependencias nativas extra.
